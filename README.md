# OS/161 Virtual Memory with Demand Paging

OS/161 kernel project developed for the Operating Systems course at Politecnico di Torino, academic year 2021/2022.

The original assignment text is not included in this repository. Based on the source code and the original project report, the assignment can be reconstructed as a virtual memory project for OS/161: replacing `dumbvm` with a demand-paged VM subsystem featuring TLB management, per-process page tables, a physical coremap, and file-backed swapping.

## Reconstructed Assignment

The project goal was to improve OS/161 virtual memory management by implementing:

- demand paging, so pages are loaded only on first access;
- lazy ELF segment loading, recording executable metadata without eagerly copying all segments into memory;
- per-process page tables;
- a physical coremap to track free and busy RAM frames;
- page replacement when no physical frame is available;
- a swap file on `emu0:SWAPFILE`, with configurable size;
- explicit TLB miss handling and TLB replacement;
- read-only text segments through the TLB dirty bit;
- VM statistics for TLB faults, page faults, reloads, invalidations, and swap writes.

## Repository Layout

This repository contains the `src/kern` subtree of OS/161, not a full OS/161 checkout.

- `conf/PAGING`: main kernel configuration for the project.
- `vm/`: virtual memory implementation.
- `include/`: shared kernel headers, including `addrspace.h`, `pt.h`, `coremap.h`, `swapfile.h`, `vm_tlb.h`, and `vmstats.h`.
- `syscall/`: program loading and syscall-related changes used by the project.
- `main/`: kernel bootstrap, shutdown, and OS/161 menu code.

The `compile/*` directories are generated by OS/161 and should not be committed.

## Main Components

### Demand Paging

`syscall/loadelf.c` introduces `read_elf()`, used instead of eager ELF loading when `PAGING` is enabled. It stores segment bounds, ELF offsets, file sizes, and the opened vnode in the address space. Pages are then loaded on demand from `vm/vm.c` when a fault occurs.

### Page Faults and TLB

`vm/vm.c` implements `vm_fault()`: it classifies the fault address as text, data, or stack, checks the page table, retrieves the page from swap or the ELF file if needed, and finally inserts the translation into the TLB through `vm/vm_tlb.c`.

The text segment is treated as read-only. TLB entries for code pages have the dirty bit disabled, so writes trigger `VM_FAULT_READONLY`.

### Coremap and Replacement

`vm/coremap.c` replaces the simple physical allocation strategy used by `dumbvm` with a coremap. The coremap tracks physical frames and selects a victim to swap out when no free frame is available.

### Swap File

`vm/swapfile.c` creates and manages `emu0:SWAPFILE`. Swap space is split into page-sized slots and tracked by a `swap_map` table.

### Page Table

`vm/pt.c` implements a per-process page table with a dirty bit and segment type for each entry. The replacement logic prioritizes pages already present in swap and treats code, data, and stack pages differently.

### Statistics

`vm/vmstats.c` maintains counters for:

- TLB faults;
- TLB faults handled with a free entry or with replacement;
- TLB invalidations;
- TLB reloads;
- zero-filled page faults;
- page faults from ELF;
- page faults from the swap file;
- swap writes.

Statistics are printed during kernel shutdown in `main/main.c`.

## Local Build with Docker

The simplest way to build and smoke-test the kernel is to use the PoliTO OS/161 Docker image:

```bash
docker run --rm -v "$PWD:/work/kern:ro" \
  -e KERN_SRC=/work/kern \
  -e CONF=PAGING \
  marcopalena/polito-os161:latest \
  bash /work/kern/scripts/os161-ci.sh
```

On PowerShell:

```powershell
docker run --rm -v "${PWD}:/work/kern:ro" `
  -e KERN_SRC=/work/kern `
  -e CONF=PAGING `
  marcopalena/polito-os161:latest `
  bash /work/kern/scripts/os161-ci.sh
```

To work manually inside the container:

```bash
docker run --rm -it -v "$PWD:/work/kern:ro" marcopalena/polito-os161:latest bash
```

Then, inside the container:

```bash
cd /home/os161user/os161/src/kern
find . -mindepth 1 -maxdepth 1 -exec rm -rf {} +
tar -C /work/kern --exclude=.git --exclude=compile -cf /tmp/kern.tar .
tar -xf /tmp/kern.tar

cd /home/os161user/os161/src/kern/conf
./config PAGING

cd ../compile/PAGING
bmake depend
bmake
bmake install

cd /home/os161user/os161/root
sys161 kernel "q"
```

## GitHub Actions

The workflow in `.github/workflows/os161.yml` uses Docker to:

1. mount this repository as `src/kern`;
2. configure the `PAGING` kernel;
3. build it with `bmake depend`, `bmake`, and `bmake install`;
4. boot System/161 with `sys161 kernel "q"` as a smoke test.

The smoke test checks that the kernel can boot and shut down correctly. Heavier user programs such as `palin`, `huge`, `matmult`, or `ctest` are not run by default in CI because they can take a long time.

## Notes

- The swap file size is defined in `include/swapfile.h` as `SWAP_SIZE`.
- The fixed per-process page table size is defined in `include/addrspace.h` as `N_FRAME`.
- The project preserves the original OS/161 Harvard source license blocks in the source files.
